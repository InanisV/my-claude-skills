# MEXC 产品方案图 Skill

## 核心理念：图是讲故事的工具，不是排版练习

一张好的方案图 = **用最少的视觉元素，把一个复杂的事讲得让老板 3 分钟看完就能拍板**。

布局是手段，不是目的。不要为了好看而堆砌组件。
每一个 section 存在的理由只有一个：**回答老板脑子里的下一个问题**。

---

## Step 1: 先理故事线，再想布局

拿到用户的方案内容后，**不要急着选布局**。先问自己（或帮用户理清）这条故事线：

### 1-3-9 叙事结构

```
1 — 一句话：这个方案是什么、解决什么问题
│
├── 3 — 三个支撑论点（为什么可行 / 为什么现在做 / 凭什么我们做）
│   │
│   ├── 9 — 每个论点下 2-3 个具体论据（数据、机制、案例）
```

**老板看方案图的心理路径：**

1. **"这是什么？"** → Hero 区一句话讲清
2. **"跟我有什么关系？"** → 痛点/机会，为什么现在要做
3. **"具体怎么做？"** → 机制/流程/架构
4. **"做了会怎样？"** → 预期效果，用数据说话
5. **"有什么风险/成本？"** → 风险可控、成本可接受
6. **"那就做吧"** → 总结收尾，强化信心

**不是每张图都要覆盖全部 6 步。** 根据实际场景裁剪：
- 给老板 pitch 新方案 → 重点 1-2-3-4
- 给团队同步架构 → 重点 1-3，展开细节
- 复盘/汇报 → 重点 1-4，数据为主
- 跨团队对齐 → 重点 1-2-3，避免术语

### 实事求是原则

- **有数据就用数据**，没数据就诚实标注"预估"、"待验证"
- **不要为了好看编造 KPI**。"预期提升 15%（基于 AI快讯同类数据推算）"比"大幅提升"有说服力一百倍
- **承认不确定性是加分项**，不是减分项。"AB 测试 2 周后用数据定"比"方案 A 一定更好"更让老板信任
- 标签用 `参考值·待测算` 之类的诚实标注，别用"预计"模糊过去

## Step 2: 故事线 → 视觉区块

每个故事节点映射到一个视觉区块。**选择最直觉的表达方式，不要为了用某个布局而用它。**

| 你要讲的事 | 自然的视觉表达 | 避免 |
|-----------|--------------|------|
| "行业都这么做，但有问题" → "我们这么做，更好" | 左右对比（红 vs 绿） | 不要用流程图，对比就是对比 |
| "分三种情况，每种都不亏" | 并排卡片 | 不要用流程图，这不是步骤 |
| "四步走，形成闭环" | 横向流程 + 箭头 | 不要用并排卡片，步骤有先后 |
| "核心指标就看这两个" | 大数字 + 注释 | 不要藏在文字段落里 |
| "我们有四个别人抄不了的优势" | 特性网格 | 不要用流程图 |
| "1 个目标 → 3 个方向 → 9 个场景" | 树形/层级结构 | 不要铺平成一堆卡片 |
| "Q1做这个 Q2做那个" | 时间线 | 不要用并排卡片 |

### 布局弹药库（可混搭）

参考 `references/design-tokens.md` 获取每种布局的完整 CSS。可用布局：
- `comparison` — 左右对比
- `cards-row` — 并列卡片（2-4列）
- `flow-steps` — 流程步骤（带箭头）
- `kpi-highlight` — 大数字指标
- `feature-grid` — 特性网格（可有高亮行）
- `hierarchy` — 层级结构 / 1-3-9 树
- `timeline` — 时间线
- `matrix` — 多维矩阵表格

**一张图通常 4-6 个 section 最佳。** 少于 3 个太单薄，多于 7 个太冗长。

## Step 3: 写标题的技巧

标题是整张图最重要的文本。老板可能只看标题就决定要不要细看。

### Section 标题公式

每个 section 标题要**自带结论**，不只是描述内容：

| ❌ 描述型（没信息量） | ✅ 结论型（一看就懂） |
|---------------------|---------------------|
| 背景分析 | 行业在事后救济，我们在**事前免疫** |
| 方案效果 | 三种结局，平台**全赢** |
| 执行流程 | 四步闭环，从**亏损**到**转化** |
| 核心指标 | 虚拟敞口 → 平台**核心 KPI** |

**关键词高亮规则：**
- 用 `<em>` 标签包裹关键词，CSS 会自动渲染为金色
- 也可以用 `.text-green` `.text-blue` `.text-red` 类做语义色
- 一个标题最多高亮 1-2 个词，多了就没重点

### Hero 标题

- 方案名 + 核心动词，控制在 10 个字以内
- 用渐变金色高亮动词部分
- 副标题补充"什么场景 · 什么方式 · 什么目标"

## Step 4: 生成 HTML

### 技术规范
- 单文件 HTML，CSS 用 `<style>` 内联
- 固定宽度 `820px` 居中（适配截图和嵌入 PPT）
- **必须先阅读 `references/design-tokens.md`** 获取完整 CSS 变量和组件样式
- 图标用 emoji（简单直觉），不用外部资源
- 中文字体栈：`"PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif`

### HTML 骨架

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <style>
    /* CSS 变量 — 从 design-tokens.md 复制 */
    /* 全局基础样式 */
    /* 各组件样式 — 只引入实际用到的 */
  </style>
</head>
<body>
<div class="page">
  <!-- Hero: 一句话讲清"这是什么" -->
  <div class="hero">...</div>
  
  <!-- Info Bar: 关键标签速览 -->
  <div class="info-bar">...</div>
  
  <!-- Sections: 按故事线展开，每个 section 回答一个问题 -->
  <div class="section">
    <div class="section-header">
      <span class="section-num">1</span>
      <h2>结论型标题，关键词用 <em>强调色</em></h2>
    </div>
    <!-- 选择最匹配的布局 -->
  </div>
  
  <!-- Footer: 一句话总结 + 核心标签 -->
  <div class="footer-bar">...</div>
</div>
</body>
</html>
```

### 转 PNG（可选）

```javascript
const { chromium } = require('playwright');
const browser = await chromium.launch({ headless: true, args: ['--no-sandbox'] });
const ctx = await browser.newContext({ viewport: { width: 820, height: 600 }, deviceScaleFactor: 2 });
const page = await ctx.newPage();
await page.goto('file:///path/to/output.html', { waitUntil: 'networkidle' });
const bbox = await page.locator('body').boundingBox();
await page.screenshot({
  path: 'output.png',
  clip: { x: 0, y: 0, width: Math.ceil(bbox.width), height: Math.ceil(bbox.height) }
});
await browser.close();
```

## 出图前检查清单

### 故事层（最重要）
- [ ] 每个 section 能回答一个具体问题，不是为了凑数
- [ ] 标题自带结论，不只是描述内容
- [ ] 数据诚实标注来源和可信度
- [ ] 不确定的指标标注"参考值·待测算"而非伪造
- [ ] 整体叙事自顶向下、逻辑通顺，删掉和主线无关的内容

### 视觉层
- [ ] Hero 标题醒目（36-40px），关键词金色渐变
- [ ] Section 编号递增，色彩一致
- [ ] 对比用红/绿色彩编码，一眼分清正负
- [ ] KPI 大数字 ≥36px
- [ ] 整体 4-6 个 section，纵向节奏均匀，留白透气
- [ ] 底栏与 Hero 呼应

## 常见方案类型速查

| 方案类型 | 推荐故事线 | 关键 section |
|---------|-----------|-------------|
| 新功能 Pitch | 痛点→方案→效果→优势 | 对比 + 流程 + KPI + 特性 |
| 季度架构规划 | 目标→方向→场景→路径 | 层级(1-3-9) + 时间线 |
| AB测试方案 | 假设→变量→预期→执行 | 对比 + 流程 + KPI |
| 复盘汇报 | 目标→结果→归因→下一步 | KPI + 对比 + 时间线 |
| 跨团队对齐 | 全局→分工→接口→节奏 | 层级 + 流程 + 矩阵 |
