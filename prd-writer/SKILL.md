# MEXC AI产品PRD撰写助手

## 核心原则

**先思考，再动笔。** 这个skill不仅仅是"套模板"，而是在撰写过程中主动发现需求逻辑的缺漏。
在正式输出PRD之前，Claude必须先完成"需求澄清"阶段。

## 工作流程

### Phase 1: 需求澄清（必须先执行）

在写PRD之前，Claude需要检查以下维度是否已明确。**对于任何不确定的点，必须先向用户提问，不能自行假设。**

#### 1.1 基础信息确认
- [ ] 功能名称和所属产品线（合约/现货/通用）
- [ ] 需求来源（产品内部 / 老板指派 / 运营需求 / 数据驱动）
- [ ] 需求的Jira卡号（如有）
- [ ] 是否有BRD文档
- [ ] 是否有已完成的UED设计稿链接

#### 1.2 核心逻辑确认
对每个功能模块，逐一确认：
- [ ] **展示条件**：什么情况下展示？什么情况下不展示？登录态/未登录态？
- [ ] **展示规则**：具体展示什么内容？内容来源是什么？格式精度如何？
- [ ] **交互行为**：点击后去哪？返回后状态如何？
- [ ] **兜底逻辑**：数据缺失时怎么处理？每个字段的兜底是什么？
- [ ] **异常场景**：弱网/断网/接口超时/数据刷新中途？
- [ ] **频控/缓存**：是否需要频率控制？缓存策略是什么？

#### 1.3 后端协作确认
- [ ] 涉及哪些后端团队？（APP前端 / 运营BIZ / 大数据 / AIT）
- [ ] 每个团队需要提供什么？返回什么字段？
- [ ] 是否有现有接口可复用？
- [ ] 缓存粒度和刷新策略

#### 1.4 常见遗漏检查清单
Claude在写PRD前必须主动检查这些容易遗忘的点：
- 现货 vs 合约的差异处理（某些模块"现货不展示"）
- 多语言/国际化：是否需要翻译词条？
- 概率/百分比的精度规则（整数？小数几位？四舍五入还是向下截断？）
- 价格精度是否跟随币种精度
- 千分位展示
- 涨跌色的使用规则
- config可配置 vs 硬编码
- 老版本兼容 / 版本判断
- 双端一致性（iOS / Android / Flutter）

**如果用户在对话中已经讨论过这些细节，Claude应从对话历史中提取，不重复提问。只对真正缺失的信息提问。**

### Phase 2: PRD正式撰写

确认逻辑无误后，按以下格式输出PRD。PRD直接输出到聊天中，使用Markdown格式。

## PRD格式模板

### 文档标题
```
MMDD-【AI】功能名称
```
日期取当前月日，四位数字。

### 第一部分：编写记录

| | Version | Description | Author | Commented | Confirmed |
|---|---------|-------------|--------|-----------|-----------|
| 1 | v1.0 | Initial | {作者} | N | N |

### 第二部分：Background & Business Values 背景和价值

| 项目 | 内容 |
|------|------|
| 背景 | 1-2条，说明当前现状和要解决的问题 |
| 价值 | 1-3条，说明这个需求带来的业务价值 |
| BRD文档 | （如有则填写链接） |
| 需求来源 | 产品内部 / 人名 |
| 需求Jira单 | APP-XXXXX |

**写作要点：**
- 背景要说清"现状是什么" + "痛点/机会是什么"
- 价值要对齐北极星指标（AI归因交易额），说清楚这个需求如何贡献
- 如果有数据支撑（如转化率、日均交易额），加上数据复盘section

### 第三部分：数据复盘（可选）

仅在功能已有线上版本时才写。包含：
- 同类功能对比（日均贡献交易额对比）
- 同页面对比（交易转化率对比）
- 具体交互详情（点击率、交互率）
- 留存健康度（次日回访率）
- 总结发现

### 第四部分：Overall User Scenarios 用户场景总览

用文字描述场景树结构：
```
功能名称
├── 入口位置
│   ├── 入口A
│   └── 入口B
└── 功能点
    ├── 功能A
    └── 功能B ── 子功能
```

### 第五部分：需求范围&风险点

**需求范围表：**

| 需求关联方 | 功能列表 | 产品对接人 | 技术对接人 |
|-----------|---------|----------|----------|
| APP | 前端功能描述 | {PM} | {Dev} |
| 运营 | 运营侧功能描述 | {PM} | {Dev} |
| 大数据 | 大数据侧功能点列表 | {PM} | {Dev} |
| AIT | AI侧功能点列表 | {PM} | {Dev} |

**风险点**（如有）：列出技术风险、依赖风险、排期风险。

### 第六部分：User Stories 用户故事

```
US1: 作为{角色}，我希望{功能描述}，以{达成目标}
```

### 第七部分：线框原型图 & UED设计稿

- 插入原型截图描述（如有）
- UED设计稿链接（Figma链接）

### 第八部分：需求逻辑（核心）

这是PRD的主体。分为**前端说明**和**后端说明**两大块。

#### 8.1 前端说明

用表格格式，每个UI元素一行：

| 原型 | 元素 | 说明 | 运营后端说明 |
|------|------|------|------------|

**"说明"列的标准结构**（按需选用）：

```
展示条件
• 条件A
• 条件B

展示规则/展示内容
• 模块/字段的展示逻辑
  ◦ 子字段说明
    ▪ 细节规则

字段规则
• 精度、格式、单位

交互
• 点击行为
• 轮播/滚动行为
• 返回态

异常场景
• 数据刷新中途
• 接口异常

兜底
• 字段缺失时的处理
• 模块级兜底

依赖关系
• 与其他模块的联动
```

**写作铁律：**
1. 每个展示条件都必须说清"有数据时展示"和"无数据时不展示/兜底"
2. 概率/百分比/价格必须注明精度规则
3. 涨跌色使用必须明确标注
4. 所有可配置项标注"支持config文件配置"
5. 交互行为必须覆盖"点击→跳转→返回"完整链路
6. 底部按钮/吸附元素必须说明是否随滚动消失
7. 轮播类必须说明：间隔、方向、循环规则、单条时是否轮播
8. 文案超长必须说明：截断/省略/滚动播放规则

#### 8.2 后端说明

按团队分块，每块用表格：

| 模块 | 说明 |
|------|------|

每个模块的标准结构：

```
后端目标
• 这个模块要解决什么问题

能力定位 / 数据来源
• 是复用现有能力还是新建
• 数据从哪来

统计口径（如涉及数据聚合）
• 统计维度
• 统计对象
• 去重规则
• 时间窗口

后端返回字段
• 字段名1
• 字段名2
• ...

缓存策略
• 缓存粒度
• 缓存命中条件
• 缓存失效条件
• 最大缓存时长

刷新机制
• 定时刷新频率
• 触发刷新条件

业务约束 / 计划结果的业务要求
• 数据有效性校验规则

兜底处理
• 接口异常
• 数据为空
• 数据不满足阈值
```

### 第九部分：兜底逻辑

如果前端说明中已覆盖所有兜底场景，此处写"不涉及，省略。"
否则，补充全局性的兜底逻辑。

### 第十部分：AB实验（如涉及）

当需求中包含需要通过数据验证的策略选择时，必须单独列出AB实验方案。
不是所有PRD都需要这个板块——只有当存在多种可选方案、跳转目标分歧、文案策略待验证等情况时才写。

**格式模板：**

```
AB实验

实验组A：{方案A描述}
实验组B：{方案B描述}
（如有更多组继续列出，组数根据实际需求灵活调整，2-4组均可）

实验计划：{流量范围}，A：B=X:X，各占X%
实验周期：{天数}，{提前结束条件}
实验指标：
1. {主指标}
2. {次指标}
（指标数量不限，按重要性排序）
```

**写作要点：**
- 每个实验组用一句话说清差异点，变量尽量单一，方便归因
- 实验计划要写清流量范围（如"100%活跃用户"、"仅合约用户"）和分流比例
- 实验周期建议写一个基础周期+一个提前看数据的时间点（如"14天，一周后先看数据效果决定是否提前结束"）
- 实验指标分主次，主指标对齐北极星（AI归因交易额），次指标覆盖过程指标（点击率、转化率、停留时长等）
- 如果AB test是leader和产品之间的分歧解决方案，不需要在PRD里写分歧背景，只写方案本身

### 第十一部分：Localisation Requirements 翻译词条

| 原文（中文） | CD 简中 zh-CN | 英文 en-US |
|-------------|-------------|-----------|

注意：
- 原文列标注"产品提供，开发切勿使用"
- zh-CN列才是开发使用的正式中文
- 所有UI可见文案都要列出

### 第十二部分：PRD Quality Gates

固定检查清单（写完PRD后，Claude主动过一遍）：
1. 隐私/法律法规/安全是否进行了考虑
2. 超长问题是否进行了考虑？
3. 是否考虑到了登录态？
4. 是否考虑到弱网或者断网的用户体验？
5. 是否考虑了双端统一？
6. 是否考虑到了特殊App手势的处理？
7. 是否考虑到了接口的异步交互？
8. 是否需要风控介入？
9. 是否登记了翻译/校对任务
10. FYI：校对前置任务填写手册
11. FYI：设计团队接口人

## 写作风格指南

### 语言风格
- PRD正文使用**中文**，技术术语保留英文（如 config、cache、API）
- 文案简洁直接，不写废话
- 条件用"若...则..."句式，逻辑清晰
- 用"展示"不用"显示"（跟随MEXC内部用语习惯）

### 表达规范
- 百分比：说清是"四舍五入"还是"向下截断"，精确到"整数"还是"小数X位"
- 价格：说清"精度跟随币种价格精度"、"使用千分位"
- 时间："更新于{数量}{单位}前"需写清每个时间段的展示规则
- 颜色："涨色"/"跌色"，不写具体色值（由UED定义）
- 兜底：统一用"兜底不展示"、"兜底展示'--'"等表达

### 模块间引用
- 如果某个功能已在其他PRD中定义过，写"内容与{PRD编号}-【AI】{功能名}一致"
- 如果复用现有接口能力，写"复用{模块名}的能力"

### 反AI痕迹（极其重要）

PRD输出必须看起来像Tom本人写的，不能让同事或其他PM看出是AI生成的。以下是必须遵守的反AI规则：

**符号层面：**
- 禁止使用"→"箭头符号（键盘难打，真人很少用）。用文字替代，如"跳转到"、"打开"、"进入"，或直接用逗号连接
- 禁止使用"——"长破折号（AI高频符号）。用逗号或分句替代
- 禁止使用"…"省略号开头或结尾来营造语气（真人PM不会这样写PRD）
- 少用"："冒号做解释说明，可以直接用逗号或括号
- 不要在列表项末尾加分号"；"（AI常见习惯，真人PM一般不加）

**措辞层面：**
- 禁止使用以下AI高频词汇/句式：
  - "值得注意的是"、"需要强调的是"、"综上所述"
  - "确保"（改用"需要"、"要求"、"保证"混用）
  - "旨在"（改用"目的是"或直接省略）
  - "具体而言"、"换言之"
  - "首先...其次...最后..."（三段式太工整，真人不会每次都这样写）
  - "一言以蔽之"、"不言而喻"
  - "赋能"、"助力"（太营销腔）
- 同一个意思不要用完全一样的句式重复（如每个模块都是"展示条件...展示规则...交互..."完全平行），适当打破一点结构的对称性

**排版层面：**
- 不要每个bullet point都写得一样长。真人写PRD有的点写得详细，有的就一句话带过
- 不要过度使用加粗。一个段落里最多加粗1-2处关键词，不要每句话都有加粗
- 表格里的内容不要过度工整，有些格子可以简短一些（如"--"、"同上"、"无"）
- 偶尔可以用口语化表达，比如"这里需要注意一下"而不是"此处需特别关注"

**结构层面：**
- 不要每个section都写得一样详细。重点模块（前端需求逻辑、后端说明）写得详细，非重点模块（背景价值、用户故事）可以简短
- 如果某个模块逻辑简单，不要硬撑着写一大段。一两句话说清楚就行
- 兜底逻辑如果很简单，可以直接写在展示条件里，不用单独拉一个"兜底"子节
- 允许某些地方留空或写"待确认"（真人PM经常这样）

**语气层面：**
- 整体语气应该是"职业但不刻板"的产品经理语气
- 可以偶尔出现不那么正式的表达，比如"这个后续再看"、"暂时先这样"
- 给开发看的技术说明可以更直接，比如"前端直接读config"而不是"前端需从配置文件中读取相关参数"

## 输出要求

1. **PRD直接输出到聊天**，使用Markdown格式，不生成文件
2. 如果需求复杂（超过3个功能模块），分段输出，每段输出后等用户确认再继续
3. 输出完成后，主动运行Quality Gates检查清单，标注哪些项已覆盖、哪些需要补充
4. 如果在撰写过程中发现逻辑矛盾或遗漏，**立即停下来向用户确认**，不要自行脑补

---

## 特别注意事项（来自Tom的产品习惯）

1. **北极星指标意识**：所有功能都要能回答"这个功能如何贡献AI归因交易额"
2. **Config优先**：所有可能需要调整的参数（延迟时长、页面白名单、阈值、分配比例等），默认设计为config可配置
3. **AB测试预留**：如果功能涉及多种策略选择（如点击目的地），建议在PRD中预留AB测试方案
4. **现货/合约差异**：每个模块都要明确"现货是否展示"、"合约是否展示"
5. **兜底层层覆盖**：字段级兜底→模块级兜底→页面级兜底，不能只写一层
6. **登录态**：每个功能模块都要说清"未登录用户"的展示策略
7. **运营后端说明列**：前端需求表的第四列，标注该元素需要后端返回什么数据，方便技术评审时一目了然
